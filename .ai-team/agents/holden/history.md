## Learnings
- **2026-02-24 — Pre-publish security audit patterns.** No `.gitignore` existed — `node_modules/` (5,829 files) and `dist/` (116 files) were tracked in git. Scan report files in `a11y-reports/` contain Azure AD tenant IDs, OAuth2 authorization URLs with client IDs/states/nonces, internal Microsoft Security Portal URLs, and base64-encoded screenshots of internal applications. These are runtime outputs, not source code, but would leak on push. `.a11y-patterns/` contains site-specific learned patterns including ADO test plan IDs. Test files (`error-page-and-url-rewrite.test.ts`, `html-cleanup.test.ts`) contained real-looking Azure tenant GUIDs and resource IDs from bug reports — replaced with obviously-fake placeholders. `.ai-team/team.md` contained corporate email address. `src/detection/types.ts` had hardcoded form test password `TestP@ss123` in default config — replaced with generic placeholder. Credential handling is sound: PAT and scanner credentials read from env vars (`ADO_PAT`, `A11Y_SCANNER_CREDENTIALS`), never hardcoded. No private keys, JWTs, connection strings, or base64-encoded secrets in source. `child_process.exec` in cli.ts (line 384) uses string interpolation for `open` command — low-risk command injection vector for local CLI tool. Dependencies are clean: all well-known packages (playwright, axios, chalk, commander, yaml, axe-core).
- **2025-07-14 — Packaging architecture assessment.** Evaluated five distribution channels (CLI, GitHub Action, ADO Pipeline Task, VS Code Extension, MCP Server). The codebase is already well-structured for a layered approach: Core engine (npm) → CLI (Commander) → thin wrappers for each channel. Playwright's browser dependency is the primary constraint — works great in CI, problematic in desktop/agent hosts. Recommended shipping order: CLI first, then ADO Task + GitHub Action in parallel, MCP Server later, VS Code Extension deferred. Key architectural principle: wrappers call the CLI (or import Core for MCP), no business logic in wrappers. `ScanResult` JSON is the interchange contract.
- **2025-02-23 — POC scaffolding and type system reconciliation.** Discovered the team had already built substantial code in parallel: Naomi (engine, crawler, page-analyzer, ADO client), Alex (CLI types, config loader, JSON/CSV/HTML reporters), Drummer (rule types + catalog), Bobbie (UI detector, flow analyzer, interaction simulator, detection types). Multiple type systems exist: scanner/types.ts (engine), types/scan-types.ts (CLI/reporting), config/schema.ts (user-facing config). Rather than creating a fourth, I aligned new interfaces (IRuleRunner, IFlowAnalyzer, IReporter, IBugCreator) with existing types. Barrel re-exports in types/ provide convenience imports without duplication. The three config layers (user-facing, engine, detection) are a valid separation of concerns — don't force-unify them.
- **2025-02-23 — Pre-existing build errors in detection/.** Bobbie's page.evaluate() callbacks have TypeScript strict-mode issues (implicit any in recursive buildSelector, unknown type on Array.from filter callbacks). These are cosmetic — the runtime code works. Bobbie should fix these before we ship, but they don't block POC.
- **2025-07-16 — Severity enum reconciliation (build fix).** Fixed 29 TypeScript compilation errors across 5 files caused by stale severity values. The canonical Severity type in `rules/types.ts` is `'critical' | 'serious' | 'moderate' | 'minor'`, but code across ado/, reporting/, scanner/, config/, and types/ was using the old values `'major'` and `'advisory'`. Mapped `major` → `serious`, `advisory` → `moderate` in: `ado/client.ts` (SEVERITY_MAP + priority logic), `ado/test-case-importer.ts` (criterionMap), `reporting/formats/html-reporter.ts` (colors, backgrounds, CSS classes, sort ranks), `scanner/hybrid-scanner.ts` (bySeverity record), `types/finding.ts` (severityToPriority map), `config/loader.ts` (validation array). Also fixed 6 semantic mismatches in `scanner/page-analyzer.ts` inside page.evaluate() callbacks (not caught by TS due to any[] return type, but wrong at runtime). Build now compiles clean with zero errors. Lesson: when a shared type union changes, grep the entire src/ tree — parallel agent work creates drift that TS only catches at typed boundaries, not inside evaluate() strings.

- **2025-07-17 — Element prioritization + ADO test plan integration design.** Diagnosed the core deep-explorer problem: `INTERACTIVE_SELECTORS` includes `nav *`, `[class*="sidebar"] *`, `[class*="CommandBar"] *` — DOM-order iteration means the first 50 elements (the per-state cap) are almost all shell chrome. Content tabs, data rows, and action buttons never get explored. Designed a three-tier priority system (P1 Content → P2 Navigation → P3 Chrome) with a three-pass classification: ARIA landmark detection → containment test → heuristic fallback. New `ElementPrioritizer` class runs inside `page.evaluate()` for performance. Also designed ADO test plan guided scanning architecture. Key finding: substantial ADO infrastructure already exists (`ado/types.ts`, `test-case-importer.ts`, `hybrid-scanner.ts`) but is not wired to the CLI or the deep explorer. Designed a `GuidedExplorer` that executes test plan steps sequentially and runs `DeepExplorer` from each step's end state — combines the test plan's known-important paths with auto-discovery of surrounding content. Added optional `LLMStepInterpreter` for natural language step parsing (gated behind `--ai-steps`). Three input methods: ADO API (`--test-plan <id>`), file (`--test-plan-file`), inline (`--steps`). Full design doc at `.ai-team/decisions/inbox/holden-ado-test-plan-design.md`. Implementation order: prioritizer first (independent value), then CLI flags + guided explorer, LLM interpreter last.
- **2026-02-23 — AI Test Plan Learning & Generation architecture.** Designed the LEARN → INVENT pipeline: after GuidedExplorer executes human test plans, a `PatternExtractor` analyzes execution traces (page snapshots, navigation flow, element coverage) and produces a `LearnedPatterns` model. A `TestPlanGenerator` then uses five strategies to create new `GeneratedTestScenario[]` that extend `ImportedTestScenario` — seamless integration with the existing execution pipeline. Four strategies are pure heuristic (coverage completion, depth completion, cross-page transfer, element type coverage); one is LLM-powered (edge case generation, gated behind `--ai-generate`). Patterns persist in `.a11y-patterns/` per-site with merge-on-save for accumulative learning across runs. Key design constraint: generated scenarios are the same type as human scenarios — no special execution path needed. The pipeline wires into `ScanEngine.execute()` as a post-guided-exploration phase: extract → generate → re-execute through the same `GuidedExplorer`. CLI flags: `--learn` (extract only), `--generate` (create + execute), `--ai-generate` (enable LLM edge cases). Seven implementation phases, ~4 weeks total, with types/database/extractor/generator parallelizable. Full design at `.ai-team/decisions/inbox/holden-ai-test-generation.md`.
