## Learnings
- **2025-07-14 — Packaging architecture assessment.** Evaluated five distribution channels (CLI, GitHub Action, ADO Pipeline Task, VS Code Extension, MCP Server). The codebase is already well-structured for a layered approach: Core engine (npm) → CLI (Commander) → thin wrappers for each channel. Playwright's browser dependency is the primary constraint — works great in CI, problematic in desktop/agent hosts. Recommended shipping order: CLI first, then ADO Task + GitHub Action in parallel, MCP Server later, VS Code Extension deferred. Key architectural principle: wrappers call the CLI (or import Core for MCP), no business logic in wrappers. `ScanResult` JSON is the interchange contract.
- **2025-02-23 — POC scaffolding and type system reconciliation.** Discovered the team had already built substantial code in parallel: Naomi (engine, crawler, page-analyzer, ADO client), Alex (CLI types, config loader, JSON/CSV/HTML reporters), Drummer (rule types + catalog), Bobbie (UI detector, flow analyzer, interaction simulator, detection types). Multiple type systems exist: scanner/types.ts (engine), types/scan-types.ts (CLI/reporting), config/schema.ts (user-facing config). Rather than creating a fourth, I aligned new interfaces (IRuleRunner, IFlowAnalyzer, IReporter, IBugCreator) with existing types. Barrel re-exports in types/ provide convenience imports without duplication. The three config layers (user-facing, engine, detection) are a valid separation of concerns — don't force-unify them.
- **2025-02-23 — Pre-existing build errors in detection/.** Bobbie's page.evaluate() callbacks have TypeScript strict-mode issues (implicit any in recursive buildSelector, unknown type on Array.from filter callbacks). These are cosmetic — the runtime code works. Bobbie should fix these before we ship, but they don't block POC.
- **2025-07-16 — Severity enum reconciliation (build fix).** Fixed 29 TypeScript compilation errors across 5 files caused by stale severity values. The canonical Severity type in `rules/types.ts` is `'critical' | 'serious' | 'moderate' | 'minor'`, but code across ado/, reporting/, scanner/, config/, and types/ was using the old values `'major'` and `'advisory'`. Mapped `major` → `serious`, `advisory` → `moderate` in: `ado/client.ts` (SEVERITY_MAP + priority logic), `ado/test-case-importer.ts` (criterionMap), `reporting/formats/html-reporter.ts` (colors, backgrounds, CSS classes, sort ranks), `scanner/hybrid-scanner.ts` (bySeverity record), `types/finding.ts` (severityToPriority map), `config/loader.ts` (validation array). Also fixed 6 semantic mismatches in `scanner/page-analyzer.ts` inside page.evaluate() callbacks (not caught by TS due to any[] return type, but wrong at runtime). Build now compiles clean with zero errors. Lesson: when a shared type union changes, grep the entire src/ tree — parallel agent work creates drift that TS only catches at typed boundaries, not inside evaluate() strings.
- **2025-07-15 — POC readiness audit.** Performed full inventory of src/ and assessed every P0 feature against PRD. Key findings: (1) The core scan pipeline (URL → crawl → analyze → report) is WIRED and functional end-to-end via cli.ts → ScanEngine → Crawler → PageAnalyzer → Reporter. TypeScript compiles clean. (2) PageAnalyzer has 6 hardcoded checks covering WCAG 1.1.1, 1.3.1, 1.4.3, 2.4.4, 3.1.1 — enough for a POC demo. (3) The RuleRunner + AccessibilityRule catalog system exists as types/interfaces but the catalog itself is EMPTY (Drummer's generation failed with 503). RuleRunner.runSingle() throws "not implemented". (4) Bobbie's detection module (UIDetector, FlowAnalyzer, InteractionSimulator, SiteMapper) is fully built but NOT wired into the scan engine — ScanEngine calls PageAnalyzer directly, never touches detection/. (5) ADO bug filing is built (AdoClient, BugCreator) but CLI just prints a warning placeholder. (6) Two config systems exist: config/schema.ts (user-facing ScanConfig) vs scanner/types.ts (engine ScanConfig) — different shapes but the CLI maps between them correctly in cli.ts lines 190-195. (7) package.json bin is "a11y-scan" but user expects "smart-a11y-scanner" — needs alias or renaming. The critical path to demo-ready is: fix bin name, verify Playwright installs browsers, and run a live scan. Everything else (rules catalog, detection integration, ADO wiring) is P1 for post-demo.
